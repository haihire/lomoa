## Workspace Agent Rules (Common)

이 파일은 루트 공통 원칙만 다룹니다.

영역별 상세 규칙:

- 서버: `server/AGENTS.md`
- 클라이언트: `client/AGENTS.md`

우선순위: 공통 규칙을 기본으로 적용하고, 영역별 파일의 규칙이 더 구체적일 경우 영역별 규칙을 우선 적용합니다.

공통 테스트 원칙:

- 테스트 이름과 assertion은 1:1로 대응합니다.
- 실제 런타임 호출 경로를 spy/mock 합니다.
- 목 데이터는 타입 계약을 지키고, 테스트마다 mock 상태를 복구합니다.
- 모호한 존재 확인만으로 통과시키지 않고, 동작/결과를 명시적으로 검증합니다.
- 에러 체크나 테스트 실패 분석 시에는 테스트 출력과 함께 `server/logs/`, `client/logs/`의 오늘치 `app-*.log`와 필요 시 `error-*.log`를 먼저 확인합니다.

공통 브랜치 운영 원칙:

- 큰 범위 작업은 상위 브랜치를 먼저 생성합니다. 예: `fix/admin`
- 세부 작업은 상위 브랜치에서 하위 브랜치로 파생합니다. 예: `fix/admin-sites-cache`
- 하위 브랜치는 먼저 상위 브랜치로 머지하고, 상위 브랜치의 작업이 충분히 쌓일 때까지 `main`으로 머지하지 않습니다.
- 즉시 `main`으로 보내는 단일 브랜치 전략보다 상위/하위 계층 전략을 기본값으로 사용합니다.

확정된 상위 브랜치 구조 (고정):

- `main`
- `fix/admin`
- `feat/admin`
- `fix/mainPage`
- `feat/mainPage`
- `chore/docs`

운영 방식:

- 세부 작업 브랜치는 반드시 위 상위 브랜치 중 하나에서 파생합니다.
- 하위 브랜치 PR의 기본 base는 해당 상위 브랜치입니다.
- 상위 브랜치 작업이 충분히 누적되기 전에는 `main`으로 직접 머지하지 않습니다.
- 사용자 명시 승인 없이 `main` 대상 PR을 생성하지 않습니다.

에이전트 작업 시작 규칙:

- 파일 수정을 시작하기 전, 에이전트는 반드시 아래를 순서대로 제시합니다.

1. 현재 요청의 작업 분류 리스트
2. 추천 브랜치 리스트(상위 브랜치/하위 브랜치)
3. "이 브랜치 구조로 진행할까요?" 확인 질문
4. `main` 대상 PR이 필요한 경우 "main으로 PR 생성해도 될까요?" 사전 승인 질문
