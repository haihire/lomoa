## 문제 1. YouTube API 일일 할당량 소진

**문제**
YouTube Data API v3는 하루 10,000 유닛이 무료 한도이며, `search.list` 한 번에 100 유닛이 소모된다.
서버 재시작 시마다 `OnModuleInit`에서 무조건 API를 호출하고, 크론(`@Cron(EVERY_HOUR)`)도 매시 재시도하여
개발 중 재시작 반복만으로 하루 할당량이 빠르게 소진됨.

**고민**

- 단순히 TTL을 늘리는 것은 콘텐츠 신선도와 트레이드오프
- 서버가 재시작되더라도 이전 캐시가 살아있다면 API를 호출할 이유가 없음
- 할당량이 소진된 상태에서 매시 재시도하면 에러 로그만 쌓이고 자동 복구가 되지 않음

**해결**

1. `OnModuleInit` 시작 시 Redis에 `youtube:videos:page:first` 키가 존재하면 API 호출 없이 즉시 반환
2. 할당량 초과(`quotaExceeded`) 감지 시 Redis에 `youtube:quota_exceeded` 키를 TTL과 함께 설정
   - TTL = YouTube 할당량 리셋 시각(매일 07:00 UTC = KST 16:00)까지 남은 초
3. 크론과 사용자 요청 모두 해당 키가 존재하면 API 호출 없이 차단, 리셋 후 자동 재개

**결과**

- 서버 재시작 횟수에 관계없이 캐시 유효 시간 동안 API 호출 0회
- 할당량 초과 후 수동 개입 없이 KST 16:00에 자동 재개
- 하루 최대 소비 유닛: 크론 24회 × (search.list 100유닛 + videos.list ~20유닛) ≒ 2,880유닛 → 10,000 한도의 28%

## 문제 2. NestJS 빌드 결과물이 생성되지 않음

**문제**
`tsconfig.json`에 `"module": "nodenext"`, `"moduleResolution": "nodenext"`가 설정되어 있었고,
`nest build` 실행 시 에러 없이 종료되지만 `dist/` 폴더에 파일이 생성되지 않음.
결과적으로 `node dist/main` 실행 시 `Cannot find module` 에러 발생.

**고민**

- ESM(`nodenext`)이 CommonJS보다 현대적이나, NestJS는 `emitDecoratorMetadata`(DI 핵심)와
  생태계 패키지 대부분이 CommonJS 기반으로 배포되어 실질적으로 사용 불가
- NestJS CLI가 내부적으로 SWC 컴파일러를 시도하다가 조용히 실패하는 케이스 존재

**해결**

1. `tsconfig.json`: `"module": "nodenext"` → `"commonjs"`, `moduleResolution` 항목 제거
   (`module: "commonjs"` 설정 시 TypeScript가 resolution을 자동 결정)
2. `nest-cli.json`: `"builder": "tsc"` 명시로 SWC 시도 없이 tsc 직접 사용
3. `deleteOutDir: true` 제거 — watch 모드에서 dist 삭제 후 재생성 타이밍 문제 방지

**결과**

- `nest build` 및 `nest start --watch` 모두 정상 동작
- 소스 코드는 `import/export` 문법 그대로 사용, tsc가 CommonJS로 트랜스파일

## 문제 3. 특성 빌드 분류 오염 (치신인데 치특신으로 분류)

**문제**
`loa_users` 테이블에 치명/특화/신속 특성 수치를 저장하고, 임계치 100 이상이면 "해당 스탯에 투자됨"으로 판단했다.
치명 1800 + 신속 600 + 특화 150인 캐릭터가 셋 다 100을 넘어 `치특신`으로 잘못 분류됨.
직업각인별 주력 빌드 통계가 오염되어 의미 없는 데이터가 됨.

**고민**

- 장비 아이템레벨이 높을수록 전체 특성 총량이 커지므로 절대값 기준은 레벨대마다 달라짐
- "주력 투자 스탯"을 판별하려면 전체 합산 대비 비율로 봐야 함
- 곁가지 특성(5~10% 수준)을 걸러낼 적절한 임계 비율 설정 필요

**해결**

- 절대값 임계치(100) 방식 → **전체 합 대비 15% 이상**이어야 "투자됨"으로 인정하는 비율 방식으로 변경
- 활성 스탯 기준으로 수치 내림차순 정렬 후 페어 조합(치신, 치특, 신특 등) 결정
- 비율 기준 활성 스탯이 1개뿐인 경우 수치 2위 스탯과 자동 페어링

**결과**

| 치명 | 신속 | 특화 | 특화 비율 | 이전 분류 | 수정 후 분류 |
| ---- | ---- | ---- | --------- | --------- | ------------ |
| 1800 | 600  | 150  | 6.1%      | 치특신 ❌ | 치신 ✓       |
| 800  | 800  | 800  | 33.3%     | 치특신 ✓  | 치특신 ✓     |
| 1200 | 1000 | 200  | 8.3%      | 치특신 ❌ | 치특 ✓       |

- 직업각인별 빌드 통계의 신뢰도 확보
- 15% 기준은 아이템레벨 1600~1700 구간에서 "의도적 투자"와 "잔여 특성"을 안정적으로 구분

## 문제 4. YouTube 인기 영상 필터와 배포 결과가 로컬/EC2에서 다르게 보임

**문제**
인기 영상 목록에서 다음 문제가 동시에 발생했다.

- "로스트아크" 검색 기준 인기 영상에 기대한 영상이 누락됨
- 1시간 이상 장시간 방송이 인기 목록에 섞여 나옴
- 로컬에서는 수정한 필터가 반영된 것처럼 보이는데, EC2 배포 결과는 이전 동작처럼 보임

특히 EC2에서는 `docker compose build --build` 후에도 예전 코드가 계속 실행되어, 필터 로직이 정상 반영되지 않는 혼선이 있었다.

**고민**

- 누락 영상 문제는 검색 결과 수집량 부족일 가능성이 높았음
- 장시간 방송 노출은 인기 목록만 다른 조건으로 필터링되고 있을 가능성이 있었음
- 배포 문제는 Docker 이미지 캐시 문제가 아니라, 컨테이너가 어떤 빌드 산출물을 복사하는지부터 확인해야 했음
- 로컬/프로덕션 결과 차이는 Redis 캐시와 빌드 산출물(`dist`) 불일치 둘 다 원인일 수 있었음

**해결**

1. 인기 영상 수집 시 페이지네이션을 4페이지(최대 200개)까지 확장했다.
2. 1시간 이상 영상 제외 조건을 인기 목록 전용 예외 없이 전체 필터에 공통 적용했다.
3. EC2 `server/Dockerfile`이 컨테이너 내부 빌드가 아니라 **호스트의 `dist/` 폴더를 그대로 `COPY`** 하는 구조임을 확인했다.
4. 배포 순서를 다음처럼 정리했다.

```powershell
git pull
cd server
npm run build
cd ..
docker compose build --no-cache nest
docker compose up -d nest
```

5. 로컬/EC2 Redis 캐시(`youtube:popular:first`)를 삭제해 이전 결과가 남지 않도록 했다.

**결과**

- 인기 영상 목록 수집 범위가 넓어져 누락되던 영상이 포함됨
- 1시간 이상 장시간 영상이 인기 목록에서 제거됨
- 배포 실패의 원인이 Docker 캐시가 아니라 **호스트 `dist/` 미빌드**였음을 확인
- EC2 운영 절차에 `npm run build` 선행이 필수라는 점을 명확히 정리함

## 문제 5. Google Sites 기반 사이트 아이콘이 일반 Google 아이콘으로 보이거나 403으로 실패함

**문제**
사이트 모음의 `낙원 스킬트리` 아이콘이 비어 있거나, 일반 Google favicon처럼 보이거나, 직접 추출한 `lh3.googleusercontent.com/sitesv/...` URL은 403이 발생하는 문제가 있었다.

같은 날 `로스트아크 인벤`, `사사게 검색기` 같은 신규 사이트도 추가하면서, 어떤 사이트는 `google s2 favicon`으로 충분하지만 Google Sites는 그렇지 않다는 점이 드러났다.

**고민**

- `site.icon` 렌더링 자체는 정상이라 데이터 소스의 icon URL 문제로 좁혀짐
- `https://www.google.com/s2/favicons?domain=sites.google.com&sz=32` 는 200이지만, 사이트 고유 아이콘이 아니라 일반 Google 아이콘만 제공함
- 브라우저에서 보이는 실제 아이콘을 쓰려면 Google Sites HTML에 들어 있는 실제 favicon URL을 찾아야 했음
- 다만 `lh3.googleusercontent.com/sitesv/...` URL은 버전마다 다르고, 일부는 외부 직접 접근 시 403이 발생함

**해결**

1. 사이트 렌더링 코드가 `site.icon`을 그대로 출력한다는 점을 확인했다.
2. `scripts/init-db.sql`과 실제 DB 값을 점검해 icon 컬럼 소스를 추적했다.
3. Google Sites 페이지 HTML에서 실제 favicon URL을 다시 추출했다.
4. 403이 나는 예전 URL 대신, 200 OK가 확인된 새 `sitesv` URL로 `낙원 스킬트리` 아이콘을 교체했다.
5. `낙원 장비`는 CodePen 특성상 직접적인 고유 favicon 추출 이점이 적어 `google s2 favicon` 방식을 유지했다.
6. 같은 흐름에서 다음 사이트를 시드 데이터에 추가했다.

- 로스트아크 인벤
- 사사게 검색기

**결과**

- `낙원 스킬트리`는 일반 Google 아이콘이 아니라 실제 사이트 favicon으로 정상 표시됨
- `scripts/init-db.sql`과 운영 DB 기준값이 일치하도록 정리됨
- 신규 사이트 2종이 설명/카테고리/icon 포함 상태로 시드 데이터에 반영됨
- Google Sites 계열 favicon은 단순 `domain` 기준 favicon API보다, 실제 페이지에서 추출한 URL 검증이 더 정확하다는 점을 확인함

## 문제 6. 좌측 카드 2개와 우측 사이트 목록의 높이 기준이 계속 충돌함

**문제**
메인 화면 레이아웃에서 다음 요구가 동시에 존재했다.

- 우측 `사이트 모음`은 콘텐츠 높이 기준으로 자연스럽게 보여야 함
- 좌측 `특성 빌드 분포` + `AI 직업 한줄평`의 총합은 우측 `사이트 모음`을 넘지 않아야 함
- `AI 직업 한줄평`은 콘텐츠가 많아져도 컴포넌트 자체가 늘어나지 않고 내부만 스크롤되어야 함

하지만 실제로는 다음 현상이 반복되었다.

- SiteList가 그리드 stretch 영향으로 하단에 빈 여백을 가진 채 늘어남
- 반대로 좌측 컬럼을 viewport 기준 고정 높이로 두면, 우측보다 길어져 레이아웃 기준이 뒤집힘
- `AI 직업 한줄평`은 `max-h`를 쓰면 내용에 따라 조금씩 커지고, `h`를 쓰면 내부 스크롤 구조가 깨지는 구간이 생김

**고민**

- CSS Grid의 `items-start`와 기본 `stretch`는 어느 쪽을 기준으로 높이를 잡을지에 직접 영향을 줌
- 좌측 컬럼 전체를 viewport 기준으로 고정하면 계산은 쉽지만, 사용자가 원하는 "기준은 사이트 모음" 요구와 충돌함
- `max-h`는 자연 높이를 유지하는 대신 콘텐츠가 늘수록 박스 자체가 커질 수 있음
- `h`는 고정 높이를 보장하지만, 내부 래퍼에 `min-h-0`, `overflow-y-auto`가 정확히 연결되지 않으면 스크롤 대신 넘침이 발생함

**해결**

이 날은 최종 단일 해법을 한 번에 확정하지 못했고, 아래 방향들을 반복 검증했다.

1. Grid를 `items-start`로 두고 SiteList를 자연 높이로 유지하는 방식
2. 좌측 컬럼을 `h-[calc(100vh-90px)]`로 고정하고 우측을 `h-full`로 맞추는 방식
3. `StatBuildList`는 1/3, `ClassSummaryList`는 2/3 비율로 나누되 gap(1rem)을 계산식에 반영하는 방식
4. `ClassSummaryList`에 `max-h` 대신 `h`를 적용하고, 내부 탭/목록에 `overflow-y-auto`를 부여하는 방식
5. `SiteList`는 자연 높이를 유지하고, 좌측 카드들은 고정 높이 + 내부 스크롤로 대응하는 방식

**결과**

- 이 이슈는 단순 클래스 한두 개 교체 문제가 아니라, **"누가 기준 높이가 될 것인가"** 에 대한 구조 문제임을 분명히 확인함
- `AI 직업 한줄평`은 고정 높이와 내부 스크롤을 분리해서 설계해야 한다는 점이 드러남
- Grid stretch, 자연 높이, viewport 고정 높이를 섞을 때 빈 여백/삐져나옴/스크롤 불능이 번갈아 발생한다는 점을 확인함
- 이후 레이아웃 수정 시에는 `SiteList 기준`, `좌측 고정 비율`, `내부 스크롤 전달 경로(min-h-0)`를 함께 점검해야 함

## 문제 7. 유튜브 인기 영상이 초반 몇 개만 보이고, 추가 탐색 UX가 끊기며 뒤쪽 영상 풀이 부족함

**문제**
하단 유튜브 인기 영상 영역에서 다음 문제가 같이 나타났다.

- 초기 로드 후 1일 전 근처 영상까지만 눈에 띄고, 뒤쪽 영상 풀이 짧게 느껴짐
- 오른쪽 버튼 또는 스크롤 끝 도달 시 추가 로드가 늦게 체감되어 멈춘 것처럼 보임
- 같은 영상을 여러 번 붙이며 React key 중복 경고가 발생할 수 있었음
- 서버는 인기 영상을 캐시하고 있었지만, 클라이언트는 이를 단계적으로 소비하지 못하고 있었음

**고민**

- 외부 YouTube API를 사용자 스크롤마다 다시 호출하면 UX는 나아질 수 있어도 할당량 관리가 나빠짐
- 가장 좋은 구조는 서버가 이미 확보한 인기 영상 풀을 Redis에 보관하고, 클라이언트는 이를 일부씩 나눠 받는 방식이었음
- 단순 append만 하면 로드 중 정지처럼 보이기 때문에, 로딩 표시와 자연스러운 등장 애니메이션이 필요했음
- 인기 영상 풀 자체가 짧으면 분할 로드를 해도 체감 개선 한계가 있으므로 서버 수집 범위도 같이 늘려야 했음

**해결**

1. `GET /api/streamers/popular`에 `offset`, `limit` 쿼리를 추가해 Redis 캐시된 인기 영상 목록을 분할 반환하도록 변경했다.
2. 클라이언트 `YoutubeList`는 초기 8개만 받고, 끝에 가까워지면 3개씩 추가 요청하도록 변경했다.
3. 추가 요청은 서버 Redis 캐시를 slice 하는 방식이라, 사용자 탐색 시 외부 YouTube API를 다시 쓰지 않도록 유지했다.
4. 중복 append 방지를 위해 `videoId` 기준 dedupe와 추가 로드 재진입 방지(ref lock)를 넣었다.
5. 목록 끝에서는 로딩 카드와 스피너를 노출하고, 새 카드에는 위에서 아래로 내려오는 enter 애니메이션을 적용했다.
6. 서버 인기 영상 수집 범위를 최대 8페이지(기존 4페이지)로 늘려 캐시 풀을 확장했다.
7. 인기 영상 검색의 `publishedAfter` 기준을 최근 3일에서 최근 7일로 확장했다.

**결과**

- 초기 로딩은 8개만 보여줘 더 가볍고, 이후 탐색은 3개씩 자연스럽게 이어짐
- 사용자 추가 탐색은 서버 캐시 기반이라 외부 API 사용량 증가를 최소화함
- 인기 영상 캐시 풀이 최대 200개 수준에서 최대 400개 수준으로 확대됨
- 최근 7일 범위까지 후보를 확보해 뒤쪽에서 더 다양한 영상이 노출됨
- 중복 key 경고와 추가 로드 시 체감 멈춤 현상이 함께 완화됨

## 문제 8. 사이트 목록이 실서버에서 기준 데이터와 어긋나고, 설명글이 간헐적으로 `??`로 노출됨

**문제**
사이트 목록 운영 중 다음 이슈가 연속으로 발생했다.

- `LOA.GG(아크그리드 최적화)` 삭제/복구 상태가 로컬 파일, 실DB, 배포 결과에서 서로 다르게 보임

- 실서버 화면(`daloa.kr`)에서 `로스트아크 인벤`, `사사게 검색기`, `KLoa` 설명이 `??` 형태로 보이는 현상이 재발

**고민**

- 단순 캐시 문제인지, DB 데이터 자체가 깨진 것인지 구분이 먼저 필요했음
- `seed_sites.mjs`는 upsert만 수행하므로, 삭제가 자동으로 반영되지 않는 구조였음
- 서버 Redis 캐시(10분)와 클라이언트 SSR 캐시(revalidate 300초)가 서로 다른 stale 창을 만들어
  DB가 정상이어도 화면은 과거 값을 보여줄 수 있었음
- 한글 깨짐은 앱 연결 문제가 아니라, 수동 SQL 실행 세션 인코딩 누락 가능성이 높았음

1. 기준 데이터 파일을 재정렬했다.

- 아크그리드 최적화(`https://aloa.gg/ko/arkgrid`) 항목을 최종 복구

2. 삭제까지 보장되는 동기화 스크립트를 추가했다.

- `scripts/sync-sites.sql`에서
  - 기준 목록 upsert
  - 기준 목록에 없는 href 삭제
    를 트랜잭션으로 수행하도록 구성

3. 실서버 반영 순서를 고정했다.

- `git push`
- EC2에서 `git pull`
- DB 동기화 SQL 실행(utf8mb4 강제)
- Redis `sites:all` 삭제
- Nest 빌드/재배포

4. 서버 측 한글 깨짐 자동복구 장치를 넣었다.

- `server/src/sites/sites.service.ts`에서 `KLoa`, `로스트아크 인벤`, `사사게 검색기`를 기준 텍스트로 정의
- 조회 시 `??` 패턴 감지 시 즉시 DB update + 응답값 보정

5. 프론트 stale SSR 캐시를 줄였다.

- `client/app/page.tsx`의 사이트 fetch를 `revalidate: 300`에서 `cache: "no-store"`로 변경
- API는 정상인데 웹페이지만 구값을 보여주는 현상을 차단

**결과**

- 실DB `loa_sites` 기준으로 아크그리드 최적화가 정상 복구되고 `is_active=1` 상태 확인

- `??` 설명 노출 재발 빈도가 크게 줄고, 재발 시 서버가 자동 복구 가능한 구조 확보
- 사이트 목록 운영 절차가 "파일 기준 동기화 + 삭제 보장 + 캐시 무효화"로 명확해짐

## 문제 11. 캐릭터 특성 빌드 API에 캐시가 없어 매 요청마다 DB 쿼리 발생

**문제**
`GET /api/characters/stat-builds`는 전체 빌드 통계를 집계하는 무거운 쿼리를 실행하는데,
캐시 없이 매 요청마다 DB를 직접 조회했다.
사용자가 페이지를 열 때마다 동일 결과를 반복 계산해 응답이 느렸다.

**고민**

- 통계 데이터는 실시간성이 크게 요구되지 않으므로, 일정 시간 동안 결과를 재사용해도 무방함
- 이미 다른 모듈(StreamersModule, SitesModule)에서 Redis 캐시를 사용 중이므로 같은 패턴 적용 가능
- 캐시 TTL은 데이터 갱신 주기(크롤러 실행 간격)보다 짧게 설정해야 fresh한 상태 유지

**해결**

1. `CharactersModule`에 `RedisModule` import 추가
2. `CharactersService` 생성자에 `@Inject(REDIS_CLIENT) private readonly redis: Redis` 주입
3. `findStatBuilds()` 실행 시:
   - Redis `characters:stat-builds` 키 히트 → JSON 파싱 후 즉시 반환
   - 캐시 미스 → DB 쿼리 실행 → 결과를 TTL 3600초(1시간)로 Redis 저장
4. 단위 테스트에 `REDIS_CLIENT` mock(`get: jest.fn().mockResolvedValue(null)`, `set: jest.fn()`) 추가, 14개 테스트 모두 통과

**결과**

- 캐시 히트 시 DB 쿼리 없이 Redis에서 즉시 반환, 응답 속도 대폭 개선
- 캐시 미스 시에만 DB 집계 쿼리 실행 → DB 부하 감소
- 기존 테스트 구조 유지, mock 추가만으로 테스트 통과

## 문제 12. 카카오 리프레시 토큰 만료 시 알림 자체가 중단되는 닭-달걀 문제

**문제**
카카오 나에게 보내기 알림은 리프레시 토큰 → 액세스 토큰 → 메시지 전송 순으로 동작한다.
리프레시 토큰이 만료되면 액세스 토큰 발급 자체가 불가하므로, 만료 알림조차 보낼 수 없는 구조였다.
토큰 유효기간은 60일이고, 수동으로 EC2 `.env`를 갱신하지 않으면 서비스 알림 기능 전체가 조용히 중단되었다.

**고민**

- 만료 이전에 미리 갱신해야 하는데, 언제 갱신할지 기준이 필요함
- Kakao는 만료 30일 전부터 토큰 갱신 API 응답에 새 리프레시 토큰을 자동으로 포함시켜 주므로, 이 시점을 활용할 수 있음
- 갱신된 토큰을 컨테이너 내부에서 `process.env`에 반영해도, 컨테이너 재시작 시 `.env` 파일이 기준이므로 파일에도 저장해야 함
- Docker `env_file`은 컨테이너 시작 시점에만 읽으므로, 실행 중 파일을 수정해도 반영되지 않음 → 볼륨 마운트로 호스트-컨테이너 파일을 공유해야 재시작 후에도 유지됨

**해결**

1. `kakao.service.ts`의 `getAccessToken()`:
   - Kakao 응답에 `refresh_token`이 포함되어 있으면 즉시 `updateRefreshToken()` 호출

2. `updateRefreshToken()` 신규 구현:
   - `process.env.KAKAO_REFRESH_TOKEN` 즉시 교체 (런타임 반영)
   - `process.env.KAKAO_REFRESH_TOKEN_ISSUED_AT` 오늘 날짜로 교체
   - `/app/.env` 파일을 정규식으로 읽어 두 항목 모두 교체 후 저장 (재시작 후에도 유지)
   - 카톡 알림 "✅ 리프레시 토큰 자동 갱신 완료 / 새 만료일: ..."

3. `checkRefreshTokenExpiry()` 크론 신규 구현:
   - `@Cron('0 0 9 * * *')` — 매일 오전 9시 실행
   - `KAKAO_REFRESH_TOKEN_ISSUED_AT` 기준 잔여일 계산
   - D-7 이내: 강제 갱신 시도 → 실패 시 수동 안내 알림("⚠️ 리프레시 토큰 만료 D-N...")

4. `docker-compose.yml` nest 서비스에 볼륨 마운트 추가:

   ```yaml
   volumes:
     - ./.env:/app/.env
   ```

   컨테이너 내부 파일 수정이 호스트에 즉시 반영되어 재시작 후에도 새 토큰 유지

5. EC2 `.env`에 `KAKAO_REFRESH_TOKEN_ISSUED_AT=2026-05-05` 수동 추가 후 컨테이너 재시작

**결과**

- Kakao가 응답에 새 토큰을 포함하는 시점(만료 30일 전 이후)부터 자동 갱신
- 매일 오전 9시 D-7 체크로 자동 갱신 실패 시에도 수동 안내 알림 수신 가능
- 볼륨 마운트로 컨테이너 재시작 후에도 갱신된 토큰 유지
- 사람이 개입하지 않아도 60일 주기로 토큰이 자동 연장되는 구조 확보

## 문제 2. 순차 처리로 인한 대기 시간 낭비

**문제**
초기 구현은 3단계가 완전히 순차적으로 실행됨:

```
1단계 전체 페이지 수집 완료
  → 2단계 전체 DB 중복 체크 완료
    → 3단계 API 요청 시작
```

수집이 끝나야만 저장이 시작되므로, 수집에 소요되는 시간이 순수 낭비.

**고민**

- 페이지를 수집하는 즉시 DB 체크를 시작하고, 신규 확인 즉시 API 요청을 보내면
  세 단계가 시간적으로 겹쳐서 실행될 수 있음

**해결**
`asyncio.Queue` 두 개를 활용한 Producer/Consumer 파이프라인으로 재설계:

```
Producer(수집) → [name_queue] → Checker × 30(DB체크) → [search_queue] → Searcher(API)
```

- Producer: loawa.com 페이지 수집 후 즉시 name_queue에 투입
- Checker: 30개 동시(`asyncio.Semaphore`) DB 중복 확인 → 신규만 search_queue에 투입
- Searcher: search_queue에서 꺼내는 즉시 LostArk API 요청 시작

**결과**

- 800 원정대 기준 수집 대기시간 약 600초가 파이프라인에 흡수되어 임계 경로에서 제거
- 첫 번째 DB 저장까지의 대기시간: 기존 600초+ → 약 30초(1페이지 수집 직후)로 단축

## 문제 3. LostArk Open API 분당 100회 제한 초과 위험

**문제**
LostArk 공식 API는 분당 100회 제한이 있으며, 초과 시 429 에러 및 일정 시간 차단.
원정대 1개 처리 시 siblings 1회 + armory 최대 30회 = 최대 31회 호출 발생.

**고민**

- 단순히 `await asyncio.sleep()`으로 딜레이를 주는 방식은 부정확하고 누적 오차 발생
- 윈도우 기반으로 실제 호출 횟수를 추적해야 정확한 제어 가능

**해결**
NestJS 서비스 내부에 이중 체크 Rate Limiter 구현:

- 모든 API 호출을 Promise 체인 직렬 큐(`apiQueue`)로 통과시켜 순서 보장
- ① 시간 체크: 1분 윈도우 경과 시 카운터 자동 리셋
- ② 횟수 체크: 80회 도달 시 잔여 윈도우 시간 + 0.5초 여유 후 자동 재개
- 한도 100회의 80%(8:2 법칙)인 80회를 실제 기준으로 적용해 안전 마진 확보

**결과**

- 장시간 운영 중 Rate Limit 초과 에러 0건
- 윈도우 리셋 후 자동 재개로 수동 개입 불필요

## 문제 4. 크롤러 실행 중 내부 상태를 알 수 없음

**문제**
초기 로그 형태:

```
[페이지 1] 수집 중... 30명 수집
[1] 유솔체 ... OK → 니나브:유솔체 (9명 저장)
[페이지 3] 수집 중... 30명 수집
```

- 수집, DB 체크, API 요청이 뒤섞여 출력되어 현재 진행 상태 파악 불가
- "API 처리 1명"이라는 수치만 보여 실제로 LostArk API가 몇 번 호출됐는지 알 수 없음
- 결과 메시지들이 계속 새 줄로 출력되어 터미널이 빠르게 스크롤됨

**고민**

- 파이프라인 구조에서 세 단계의 상태를 한눈에 보여줄 방법 필요
- Python에서 NestJS 내부 Rate Limiter 상태를 관찰할 수 있어야 함

**해결**

1. `\r`(Carriage Return)을 사용해 매초 같은 줄을 덮어쓰는 방식으로 변경
2. NestJS에 `GET /api/lostark/stats` 엔드포인트 추가 → 현재 윈도우 호출 횟수 반환
3. Python 로거가 1초마다 해당 엔드포인트를 폴링하여 실제 API 호출 횟수 표시

**결과**

```
현재페이지 7 / 대기자 120명 / API 34/80회 / 완료 45명
```

- 터미널 1줄만 사용, 스크롤 없음
- 현재 페이지 / 큐 대기 / LostArk API 실사용량 / 저장 완료 수를 실시간 확인 가능
- 80/80회 → 0/80회 전환 시 rate limit 대기 중임을 즉시 인지 가능

## 문제 5. 원정대별 DB INSERT N회 발생

**문제**
원정대 처리 시 캐릭터마다 개별 INSERT 실행:

- 원정대 15명 = 15번 DB 왕복
- 800 원정대 기준 = 12,000번 DB 커넥션 발생

**해결**
원정대 전체를 `buffer[]`에 누적 후 단일 배치 INSERT로 처리:

```sql
INSERT INTO loa_users VALUES (...),(...),(...),...
```

**결과**

- 원정대당 DB 쿼리 횟수: N회 → 1회
- 800 원정대 기준: 12,000회 → 800회 (93% 감소)

## 문제 6. 탭 전환 시 우측 패널의 스크롤 위치가 이전 상태를 기억함

**문제**
특성 빌드 분포(`StatBuildList`)와 AI 직업 한줄평(`ClassSummaryList`) 탭을 전환할 때,
우측 목록이 이전 탭에서 스크롤한 위치를 그대로 유지했다.

예: 치신 탭에서 스크롤 끝까지 내려간 후 특신 탭으로 전환하면, 특신 탭도 같은 위치에서 시작됨.

**고민**

- React의 컴포넌트 재렌더링 메커니즘: 동일한 DOM 요소(리스트)는 위치 상태(`scrollTop`)를 그대로 유지
- 각 탭이 별도 데이터를 가지고 있지만, 렌더링되는 `<ul>` 엘리먼트는 동일한 인스턴스이므로 스크롤 상태가 공유됨
- 탭 전환 시 컴포넌트를 강제 재마운트(remount)해야 하는데, 방법이 여러 가지 존재

**해결**

1. 우측 `<ul>` 엘리먼트에 `key={activeTab}` 속성 추가
2. React의 reconciliation 알고리즘이 `key` 변경을 감지하면 기존 DOM을 버리고 새로 생성
3. 새로운 DOM은 `scrollTop: 0` 초기 상태로 시작
4. 동일한 처리를 `StatBuildList`와 `ClassSummaryList` 두 컴포넌트에 적용

**결과**

- 탭 전환 시 우측 목록이 항상 맨 위(scrollTop=0)부터 시작
- 사용자가 각 탭에서 독립적으로 스크롤 가능
- `key` 속성 추가로 해결되어 추가 상태(scrollPos 변수)나 useEffect 보일러플레이트 불필요

## 문제 7. 즐겨찾기 기능 부재 및 localStorage 활용 미흡

**문제**
사용자가 유용한 사이트를 자주 방문하는데, 매 방문마다 여러 사이트 중에서 찾아야 했다.
모바일/데스크톱 환경 구분 없이 같은 기기에서는 이전 방문 기록을 유지하고 싶다는 요청 발생.

**고민**

- 북마크 기능을 구현하되, 서버 DB에 저장하지 않고 클라이언트 localStorage만 사용할지 여부
  - DB 저장 방식: 사용자 계정 필요, 복잡도 증가
  - localStorage 방식: 동일 기기/브라우저에서만 유지, 계정 불필요
- localStorage에 어떤 형식(배열 vs Set vs 객체)으로 저장할지
  - 배열: 추가/제거 순서 추적 가능, JSON 직렬화 편함
  - Set: 중복 자동 제거, 순서 보장 불가
- 즐겨찾기 항목 표시 방식(별, 하트, 핀 등)

**해결**

1. **저장소**: localStorage만 사용, 키는 `loa_favorites`
2. **데이터 구조**: 즐겨찾기 사이트 href를 배열로 관리
   - `favorites: string[]` — 추가된 순서대로 정렬
3. **UI 표현**:
   - 별 모양 아이콘(SVG)
   - 즐겨찾기 미추가: 회색
   - 즐겨찾기 추가됨: 노란색(#FBBF24)
4. **카드 스타일**: 즐겨찾기 사이트는 파란 테두리 + 파란 배경(`border-blue-400 bg-blue-50`)
5. **정렬 규칙**: 즐겨찾기 항목을 맨 위에 배치, 같은 레벨 내에서 추가 순서 유지

**결과**

- 사용자가 자주 방문하는 사이트를 별 클릭으로 상단에 고정 가능
- 새로고침/탭 전환 후에도 localStorage에서 즐겨찾기 복원
- 데이터는 로컬만 저장되어 개인정보 유출 우려 없음
- 향후 회원가입 기능 추가 시 서버 동기화로 확장 가능한 구조

## 문제 8. 방문자 통계 수집 불가

**문제**
사이트 런칭 후 방문자 수, 사용 기기 분포, 지역 분석 등 기초적인 메트릭이 없었다.
Google Analytics는 설정이 복잡하고, 간단한 카운팅만 필요한 상황.

**고민**

- Google Analytics: 설정 복잡, 쿠키 동의 필요, 개인정보보호법 고려
- Vercel 자체 분석 도구: Vercel에 배포된 프로젝트라면 최소 설정으로 활용 가능
- 기타 SaaS(Mixpanel, Amplitude): 유료 또는 무료 한도 제한

**해결**

1. `npm install @vercel/analytics@latest` 설치
2. `client/app/layout.tsx`의 RootLayout에서 Analytics 컴포넌트 import
   ```tsx
   import { Analytics } from "@vercel/analytics/react";
   ```
3. `<body>` 내부 마지막에 `<Analytics />` 추가
4. 빌드/배포 후 자동으로 Vercel Dashboard에 데이터 수집 시작

**결과**

- 추가 설정 없이 다음 메트릭 자동 수집:
  - 월별 페이지뷰
  - 기기별 분포(Desktop, Mobile, Tablet)
  - 브라우저 종류
  - 지역(국가 단위)
  - 주요 진입 경로
- 월별 50,000 이벤트 무료 한도 충분 (예상 일 방문자 600명 기준)
- Vercel Dashboard에서 실시간 조회 가능

## 문제 9. YouTube 영상 섹션의 스크롤바 미표시 및 네비게이션 불편

**문제**
YouTube 영상 목록(6개 카드)이 가로로 자동 스크롤되는 컴포넌트였으나:

1. 스크롤바가 보이지 않아 스크롤 가능 여부를 사용자가 모름
2. 좌우 네비게이션 버튼이 없어 마우스 드래그나 터치만 가능
3. 모바일에서 스크롤이 불편함

**고민**

- 수평 스크롤바를 보이게 하려면 `overflow-x-auto` 필요한데, 기본 스크롤바는 목록을 압박
- CSS 3D `rotateX(180deg)` 변환으로 역방향 스크롤바를 상단에 배치하는 시도
- JavaScript 이중 스크롤 동기화: 보이는 스크롤바와 실제 리스트 scroll 값 연동
- 버튼 위치: 절대 포지셔닝(목록 위에 오버레이)인지, 헤더 행에 인라인인지

**해결**

1. **상단 스크롤바 구현**:
   - 상단에 별도 `<div ref={topScrollRef}>` 컨테이너 배치
   - 내부에 너비를 동적으로 설정하는 `<div>` spacer 배치
   - `useEffect` + `ResizeObserver`로 리스트의 `scrollWidth`와 spacer 너비 동기화
   - 스크롤바는 Tailwind 커스텀 클래스로 스타일링 (높이 h-3, 색상 slate-400)

2. **이중 스크롤 동기화**:

   ```tsx
   topScrollRef.addEventListener("scroll", () => {
     listRef.scrollLeft = topScrollRef.scrollLeft;
   });
   listRef.addEventListener("scroll", () => {
     topScrollRef.scrollLeft = listRef.scrollLeft;
   });
   ```

   - `syncingRef`로 무한 루프 방지

3. **네비게이션 버튼**:
   - 헤더 행(`<div className="flex items-center gap-3">`)에 타이틀 옆 인라인 배치
   - 크기: `h-10 w-10`(충분히 큼)
   - 모양: 둥근 테두리(`rounded-md`) + 회색 배경
   - 상태: 로딩 중 또는 아이템 없을 시 `disabled={...}`로 비활성화
   - 스타일: `disabled:opacity-40 disabled:cursor-default`

4. **간격 최적화**:
   - 헤더 마진: `mb-3` → `mb-1`(타이틀과 스크롤바 간격 축소)
   - 스크롤바 마진: `mb-2` → `mb-1`(스크롤바와 영상 카드 간격 축소)

**결과**

- 상단에 항상 보이는 수평 스크롤바로 스크롤 가능성 명시
- 좌/우 버튼으로 마우스/터치 스크롤 어려운 사용자도 쉽게 네비게이션 가능
- 스크롤바와 버튼 동기화로 UX 일관성 확보
- 모바일에서 버튼 크기(10×10 = 약 40×40px)가 충분해 접근성 향상

## 문제 10. 페이지 하단에 저작권/서비스 정보 부재

**문제**
페이지 구조가 메인 콘텐츠만 있고 footer가 없었다.
로스트아크 공식 자산(이미지, 데이터)을 사용하고 있으므로 저작권 표시 필요.
또한 사용자에게 서비스 정보(연락처, 약관 등)를 제공할 공간 부재.

**고민**

- Footer 위치: 콘텐츠가 짧을 때 화면 하단 고정 vs 콘텐츠 아래 배치
- Footer 높이: 최소한의 정보만 표시할지, 링크/섹션이 여러 개일지
- 스타일: 메인 콘텐츠와의 시각적 구분

**해결**

1. **레이아웃 재구성**:

   ```tsx
   <div className="flex min-h-screen flex-col">
     <div className="flex-1 py-3">{children}</div>
     <footer>{...}</footer>
   </div>
   ```

   - Outer: `flex min-h-screen flex-col` — 항상 전체 높이 점유, 세로 배치
   - Main content: `flex-1` — 남은 공간 모두 점유
   - Footer: 별도 형제 요소로 하단 고정

2. **Footer 스타일**:
   - 배경: 연한 회색(`bg-slate-50`) + 블러(`backdrop-blur`) 적용
   - 테두리: 상단만(`border-t border-slate-200/80`)
   - 패딩/여백: `px-4 py-4`
   - 텍스트: 작게(`text-xs`) + 회색(`text-slate-500`)

3. **콘텐츠**:

   ```tsx
   <p className="text-[11px] text-slate-400">
     Lost Ark and related assets belong to Smilegate RPG.
   </p>
   ```

   - 저작권: Smilegate RPG 표시
   - 향후 확장 가능: 이용약관, 개인정보처리방침, GitHub 링크 등 추가

**결과**

- 로스트아크 공식 저작권 표시로 법적 준수
- 페이지 끝이 명확해 사용자 경험 개선
- 밝고 깔끔한 스타일로 메인 콘텐츠와 구분
- 향후 서비스 정보(About, Contact, Privacy) 확장 기반 마련
